# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (no shadcn) |
| Preset | not applicable |
| Component library | none — native HTML form elements (`select`/`input`/`button`) styled directly with Tailwind, matching the hand-rolled pattern already established in Phase 1's admin routes (`app/(admin)/admin/login/page.tsx`, `app/(admin)/admin/page.tsx`) |
| Icon library | `lucide-react` (not yet installed — add as a dependency this phase; needed for size-guide info icon, cart remove icon, edit/pencil icon) |
| Font | Geist Sans, `var(--font-geist-sans)` — already loaded globally in `app/layout.tsx` via `next/font/google`; storefront routes inherit it automatically, no new setup needed |

**Rationale — no shadcn init this phase:** `components.json` was not found. CLAUDE.md recommends shadcn/ui + Radix, but Phase 1 already built the admin UI with hand-rolled Tailwind instead (no shadcn used there either). shadcn initialization requires an interactive step (visiting ui.shadcn.com/create and pasting a preset string) that cannot be completed inside this research pass. To avoid blocking Phase 2, this contract specifies a hand-rolled Tailwind approach consistent with the existing admin pattern. **Recommendation for the owner:** consider initializing shadcn before Phase 5 (admin panel — order table, dialogs, badges benefit most from it); revisit for the storefront only if Phase 2's hand-rolled components prove insufficient.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, patch checkbox spacing |
| sm | 8px | Compact spacing (form field internal padding, badge padding) |
| md | 16px | Default element spacing (between form fields, cart item padding) |
| lg | 24px | Section padding (form section blocks, cart panel padding) |
| xl | 32px | Layout gaps (space between form column and cart panel on desktop) |
| 2xl | 48px | Major section breaks (page header to form, form to footer explainer) |
| 3xl | 64px | Page-level top/bottom spacing |

Exceptions:
- **44px minimum touch target** on all interactive controls used on mobile: size buttons, patch checkboxes (tap area, not just visible box), remove/edit icon buttons, the mobile cart bottom-bar itself. This is a customer-facing storefront checked on phones — undersized tap targets directly hurt conversion.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label / helper | 14px | 400 (regular) | 1.5 |
| Body | 16px | 400 (regular) | 1.5 |
| Heading (section) | 20px | 600 (semibold) | 1.2 |
| Display (page title) | 28px | 600 (semibold) | 1.2 |

Only 2 weights used across the phase: **regular (400)** for body/label/paragraph text, **semibold (600)** for headings, buttons, price totals, and selected-state emphasis. Do not introduce `font-medium` (500) — Phase 1's admin login page used it for labels/buttons; this phase intentionally consolidates to the 2-weight rule above. Labels use regular weight, not medium.

Examples of role usage in this phase:
- Display (28px/600): page hero heading, e.g. "Sett sammen draktene dine"
- Heading (20px/600): section headers — "Velg liga og lag", "Din bestilling" (cart panel title), size-guide modal title
- Body (16px/400): explainer paragraph (PROD-07), empty-state body copy, patch labels
- Label/helper (14px/400): field hint text, validation error text, size-guide table cells, price line-items (unit price, subtotal)

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` (light) / `#0a0a0a` (dark) | Page background — reuses existing `--background` variable from `app/globals.css` |
| Secondary (30%) | `#f4f4f5` zinc-100 (light) / `#18181b` zinc-900 (dark), borders `#e4e4e7` zinc-200 / `#27272a` zinc-800 | Cart panel background, form section cards, size-guide modal surface — matches Phase 1's `border-zinc-200 bg-white dark:border-zinc-800 dark:bg-zinc-950` card pattern |
| Accent (10%) | `#059669` emerald-600 (light) / `#10b981` emerald-500 (dark) | Reserved ONLY for: primary "Legg i handlekurv" button, "Gå til betaling" button, selected-state background/ring on league/team/size/patch selection controls, cart item-count badge, running total price text |
| Destructive | `#dc2626` red-600 (light) / `#f87171` red-400 (dark) | Reserved ONLY for: name/number validation error text + input border, "Fjern" (remove item from cart) control |

Accent reserved for: primary CTA buttons (add-to-cart, go-to-payment), the currently-selected option in each cascading selector (league/team/size) and checked patch chips, the cart badge showing item count, and the live total price figure. Nowhere else — do not tint decorative elements, dividers, or unselected controls with the accent color.

**Rationale:** Phase 1's admin already established a monochrome zinc/black-and-white base. This phase keeps the same dominant/secondary neutrals for brand consistency across admin and storefront, and introduces a single accent (emerald, evoking a football pitch) reserved strictly for calls-to-action and selection state — satisfying the 60/30/10 rule without inventing a competing brand palette PROJECT.md never specified.

**Primary visual anchor:** On first load, the accent-colored "Legg i handlekurv" button and the sticky cart panel's running total price are the primary visual anchors on the storefront screen; the 28px hero heading is secondary, and unselected form controls stay neutral so the eye is drawn to the action and the price.

## Component & Interaction Notes

*(Supplementary to the standard template — captures phase-specific structural decisions from `02-CONTEXT.md` that the planner/executor need.)*

### Layout & responsive behavior (D-13, D-16)
- Single page, two-column layout at `lg:` (1024px) breakpoint and above: main column = configurator form, right column = cart/order-summary panel.
- Desktop (`lg+`): cart panel is `sticky top-6`, fixed width ~360px, `bg-zinc-50 border border-zinc-200 rounded-lg`, visible from first page load (not just after first add-to-cart), scrolls independently once content exceeds viewport height.
- Mobile (`<lg`): cart panel collapses to a fixed bottom bar (`fixed bottom-0 inset-x-0 bg-white border-t border-zinc-200 shadow-sm`) showing item count badge + running total + a chevron/handle to expand into a full-height bottom sheet. Bottom bar must not overlap the form's own CTA button — reserve safe-area padding at the bottom of the scrollable form content equal to the bottom bar's height.

### Cascading form (D-14, D-15)
- Single page/form, not a multi-step wizard. Fields render top-to-bottom in one column inside the main configurator column: League/national-team → Team → Season (read-only) → Size → Patches → Name → Number → "Legg i handlekurv".
- First selector combines `LEAGUES` (by `league.name`) and a synthetic "Landslag" entry representing `NATIONAL_TEAMS` (this list has no league wrapper in `config/leagues-teams-seasons.ts` — treat it as a pseudo-league option in the first selector).
- Team selector is disabled/empty until a league (or "Landslag") is chosen; its options are exactly that league's `teams` array (or `NATIONAL_TEAMS` if "Landslag" was picked).
- **Season is not an interactive control.** `config/leagues-teams-seasons.ts` exposes a single `CURRENT_SEASON` constant ("2025/26"), not a per-team list — despite CONTEXT.md's D-15 describing a cascading season selector, the actual Phase 1 catalog has only one season value. Render season as a read-only label (e.g., "Sesong: 2025/26") that appears once a team is selected, not a dropdown. Flag this discrepancy to the planner/executor; no user decision needed to resolve it, since presenting the fixed value as read-only text satisfies D-15's spirit (invalid combinations remain structurally impossible — there's only one valid season).
- Invalid combinations are structurally prevented by disabling/filtering dependent selects, never surfaced as a validation error after the fact.

### Size guide (D-17, D-18, D-19)
- Size selector: 6 button/radio options — S, M, L, XL, XXL, 3XL (adult sizing only, no kids' sizes).
- A "Se størrelsesguide" text link with a lucide `info` icon sits inline next to the "Størrelse" label; clicking opens a modal (not inline-expanding content).
- Modal content: a placeholder table — columns "Størrelse", "Brystvidde (cm)", "Lengde (cm)" for all 6 sizes, with the size-guide disclaimer copy (see Copywriting Contract) shown above or below the table so it reads clearly as provisional.

### Patches (D-9/D-10 from Phase 1, consumed here)
- Render `PATCHES` from `config/patches.ts` as a checkbox/chip group under "Merker (valgfritt)".
- "Ingen" is mutually exclusive with the other three: selecting "Ingen" unchecks all other patches; selecting any other patch unchecks "Ingen". Default selected state when the form is fresh: "Ingen" pre-checked, so the form is valid without forcing an explicit patch decision.

### Name/number validation (D-20–D-23)
- Both fields optional; validation only runs on non-empty input, live on blur/change (not only on submit).
- Name: max 12 characters; allowed characters are letters (including æøå and other international letters), spaces, hyphens.
- Number: integer 0–99 inclusive, no decimals/negatives.
- Error copy per field is defined in the Copywriting Contract above; render inline below the field in destructive-color text (14px/400), and add a destructive-color border to the invalid input itself.

### Cart item cards (D-12)
- Each cart line shows: league/team + read-only season, size, patch list (or "Ingen"), name/number if provided, current unit price (from `getUnitPriceOre(cart.length)` — recomputed for ALL items whenever cart size changes, since pricing is whole-order tiered per `config/pricing-tiers.ts`, not per-item).
- Two actions per card: "Rediger" (pencil icon) reopens the configurator form fully pre-filled with that item's values for in-place editing; "Fjern" (trash icon, destructive color) removes it with the 5-second undo affordance described in the Copywriting Contract.

### Order summary (PROD-05)
- Live-updating on every cart mutation (add/edit/remove): quantity, current unit price at that tier, subtotal, and — only when the tier just changed favorably — a short "Pakkerabatt aktivert" note near the total, plus the final total in NOK (formatted with `kr` suffix, e.g. "870 kr").
- All pricing math imports from `config/pricing-tiers.ts` (`getUnitPriceOre`, `getOrderTotalOre`) — never reimplemented in the UI layer.
