<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anon (public PostgREST) → orders table | Untrusted public callers must never read customer PII or insert orders |
| retried webhook (Phase 3/4) → orders insert | Duplicate provider events must not create duplicate orders |
| authenticated admin session → orders | The one legitimate admin must be able to read; identity resolved by a stable email allowlist, not a fragile UUID |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | RLS policies on orders/order_items/order_status_history | mitigate | Every policy `to authenticated`, no `using (true)`, no anon INSERT policy; admin access resolved via `is_admin()` email allowlist (not a hardcoded UUID that would silently deny the real admin); anon negative-access proven by `tests/rls.integration.test.ts`, authenticated-admin positive-access proven by Plan 04 `tests/rls-admin.integration.test.ts` |
| T-02-02 | Tampering / Repudiation | orders idempotency | mitigate | Hard DB `UNIQUE(provider, provider_ref)` + `UNIQUE(provider, provider_event_id)` established day one; proven by 23505 assertion in idempotency test |
| T-02-03 | Elevation of Privilege | schema drift via dashboard edits | mitigate | Schema authored only as a git-versioned Supabase CLI migration; no dashboard SQL-editor source of truth |
| T-02-04 | Elevation of Privilege | admin_users allowlist tampering | mitigate | `admin_users` has RLS enabled with NO anon/public policy; only the service-role key (seed script) writes it; `is_admin()` is SECURITY DEFINER with a pinned `search_path` |
</threat_model>
