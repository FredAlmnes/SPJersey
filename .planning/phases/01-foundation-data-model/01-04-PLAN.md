<objective>
Build the walking-skeleton vertical slice: the owner can sign in at `/admin/login` with the single seeded admin account and reach a protected `/admin` dashboard that performs a real, RLS-scoped read against the live `orders` table. The auth gate is enforced twice per CLAUDE.md — in `middleware.ts` (session refresh + `getUser()`) and again in the admin `layout.tsx` server re-check — never via `getSession()`. A one-off `scripts/seed-admin.ts` creates the single admin user AND inserts its email into the `admin_users` allowlist (Plan 02) so the `is_admin()` RLS policy actually grants the real admin DB access (no public sign-up, ever).

Purpose: This is the end-to-end capability that proves scaffold → routing → Supabase Auth → RLS-protected DB access all work together (Phase 1 success criterion #1, ADMIN-01). Seeding the `admin_users` row is the step that closes the RLS loop: without it, the `is_admin()` email lookup would find no match and the authenticated admin would be denied exactly like anon. After this plan a real user can log in and see a dashboard backed by a real DB query, and an integration test proves the authenticated admin (not just anon) is genuinely authorized.
Output: A working, gated admin login → dashboard slice with the admin user seeded, the admin_users allowlist populated, and both auth wiring and authenticated-admin RLS read covered by integration tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| unauthenticated browser → /admin/* | Must be redirected; no admin content or DB data rendered |
| stale/forged session cookie → server auth check | Must be revalidated against the Auth server, not trusted from the cookie |
| authenticated admin JWT → orders RLS | Admin email must be in the allowlist for is_admin() to grant access; seed script is the sole population path |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-04-01 | Spoofing | admin auth gate | mitigate | `getUser()` (server-revalidated) in BOTH middleware and layout; `getSession()` forbidden — grep gate asserts its absence; proven by auth integration test |
| T-04-02 | Elevation of Privilege | public sign-up surface | mitigate | No sign-up route exists; single admin created only via out-of-band `scripts/seed-admin.ts` using the service-role key |

| T-04-04 | Denial of Service (self-inflicted) | broken admin RLS wiring | mitigate | Seed script populates `admin_users` with the exact ADMIN_EMAIL so `is_admin()` grants the real admin; `tests/rls-admin.integration.test.ts` proves the authenticated admin can actually read orders (regression guard for the silent placeholder-UUID lockout) |
